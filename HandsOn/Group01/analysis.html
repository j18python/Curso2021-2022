The analyses performed over the data source

Los datos que hemos analizado han sido obtenidos de 2 CSV.
En primer lugar tenemos los datos de Centros Educativos de Madrid:

  • Identificador: valor String de 8 cifras, empieza por 28
  • Nombre: valor de tipo String
  • Tipo: valor String, los distintos tipos que tenemos en este apartado son centros privados, públicos o concertados que pueden tener formación Infantil, Primaria, Secundaria o Bachillerato.
  • Titularidad: valor String, pueden ser [PRIVADO, PRIVADO CONCERTADO, PÚBLICO, PÚBLICO-TITULARIDAD PRIVADA]
  • Municipio: valor String de [001-183]
  • Coordenada_X: valor String de 6 cifras
  • Coordenada_Y: valor String de 7 cifras
  • Página Web: valor anyURI. Ejemplo: "http://www.w3.org/TR/xmlschema-0"
  • Dirección:
    • Nombre: valor String
    • Número: valor String. En el caso de que no tenga número toma el valor de "s/n"
    • Código Postal: valor String que empieza por 28
    • Tipo de vía: valor String

Por último, tenemos los datos de los Los Residentes de la Comunidad de Madrid por Edades:

  • Identificador: valor String de [001-183]. Es el del municipio al que pertenecen
  • Nombre: valor String. Es el del municipio al que pertenecen
  • Sexo: valor String, puede ser [Mujer, Hombre]
  • Edad mínima: valor numérico entero siendo el mínimo valor el 0
  • Edad máxima: valor numérico entero siendo el máximo valor el 199
  • Población: valor numérico entero, de [0-98949]

Resource naming strategy

Para elegir el nombre apropiado de las URIs que manejaremos en nuestra app hemos seguido las recomendaciones y pautas vistas en clase. En primer lugar hemos seleccionado como dominio base http://colegiosapp.org, a partir del cual hemos separado en diferentes endpoints la ontología y los recursos.

Así pues, a partir de /ontologia podemos encontrar las clases definidas y las propiedades. Cabe mencionar que ambas parten de este mismo punto, pues aunque primeramente se podría pensar que las propiedades deberían ser cada una a partir de una clase no formaría URIs correctas ya que no existe jerarquía entre las diferentes definiciones. Estas clases irán precedidas del separador hash #.

Tras el dominio base se ha definido la ruta /resource a partir de la cual colgarán todos los recursos definidos para este conjunto de datos. Irán precedidos por el slash /.


De esta manera, la lista completa de clases y propiedades quedará definida de la siguiente manera:

  • Dominio: http://www.colegiosapp.org
  • Ruta ontología clases: http://www.colegiosapp.org/ontology#[nombreClase]
    • Municipio: http://www.colegiosapp.org/ontology#Municipality
    • Colegio: http://www.colegiosapp.org/ontology#School
    • Dirección: http://www.colegiosapp.org/ontology#Address
    • Grupo de residentes: http://www.colegiosapp.org/ontology#ResidentGroup
  • Ruta ontología propiedades: http://www.colegiosapp.org/ontology#[nombrePropiedad]
    • Sexo: http://www.colegiosapp.org/ontology#hasGender
    • Nombre: http://www.colegiosapp.org/ontology#hasName
    • Número: http://www.colegiosapp.org/ontology#hasNumber
    • Código postal: http://www.colegiosapp.org/ontology#hasPostalCode
    • Tipo: http://www.colegiosapp.org/ontology#hasType
    • Edad máxima: http://www.colegiosapp.org/ontology#maxAge
    • Edad mínima: http://www.colegiosapp.org/ontology#minAge
    • Número de personas: http://www.colegiosapp.org/ontology#numPeople
    • Propiedad: http://www.colegiosapp.org/ontology#ownership
    • Coordenada X: http://www.colegiosapp.org/ontology#xCoordinate
    • Coordenada Y: http://www.colegiosapp.org/ontology#yCoordinate
  • Ruta recursos: http://www.colegiosapp.org/resource/[identificador]
    • Grupo de residentes: http://www.colegiosapp.org/resource/Group_X_Y_Gender
